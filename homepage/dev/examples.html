Workstyle

To get clean code we decided to create some code-"rules" for us. You find here every information about do's and don't within our code.

  1. Code practice

    • Naming

      Classes/Interfaces Every word is wrote behind each other, without seperation, big initial letter. No abbrevations should be used.

      Good example:
      class MyClass


      Bad example:
      class myClass
      class myCls
      class My_Class
      class myclass
      Packagenames Like normal Methods we write them. Packagenames should be contain one word(avoid using more), written in lowercase letters.
      A packagename should be inverse the same name like the domain where it is provided. In our case we use org.as2lib. So users won't have any problem with it.
      Packages or classes that won't get provided (like Testcases) should not be in this package.
      Good example:
      org.as2lib.basic


      Bad example:
      org.as2libBasic
      org.As2LibBasic
      Org.As2lib.Basic
      Methods/Vars Methods should be written lower case. Each word should be separated by a uppercase initial letter.

      Good example:
      private var aVar:String;
      private function aMethod ():Void { }


      Bad example:
      private var $aVar:String;
      private var AVar:String;
      private var A_Var:String;
      private var Avar:String;
      private var _aVar:String;
      Constants Constants should always be written with capitals, using an underscore to seperate. Constants should always be static & if nothing speaks against it public.
      Good example:
      public static var USE_FONT:Number;
      private static var HIDE_COLUMN:String;


      Bad example:
      private static var Use_Font:Number;
      private static var USEFONT:Number;
      public static var useFont:Number;
      private var Ignore:String;
      private var $hide:String;

    • Codeformat

      • No indents by tabs, all indents by two spaces.
      • A line should be max. about 120 chars. (try to limit it to about 100 chars)
      • Use no braces with the return statement if you have no operation.elseway use them.
        function doesSomething (a:String, b:String) {
  if(a == b) {
    return (a-b);
  }
  return a;
}
      • Braces behind methods and classes are always in the same line, using a space for seperation: 
        class A {
  function A () {
  }
}
      • Between the condition keyword and the condition should not be a character free. Seperations only between characters. 
        if(a == b && b == c) {
}
      • Braces for conditions or loops should also be within the same line: 
        function a (a:String, b:String) { 
  if(a == b) {
    // Code
  } else {
    // Code
  }
  for(var a:Number = 0; a <= 10; a++) {
    // Code
  }
  switch(a) {
    case "0":
    case "1":
    default:
  }
  
}
      • If the Arguments for an method exceeds 120 chars you should use linebreaks. 
        class A {
  function A ( anArgument:String, anotherArgument:Number, aThirdArgument:Object,
               aFourthArgument:MyClass) {
    // Code
  }
}
      • Use Void if an Method returns nothing or uses no argument. This will increase the readability.
        function doAllIWant( Void ):Void {
}

    • Good practise:

      • Don't (never!) use as2 internal Setter and Getter (call them "bad"!). You won't be able to define them within interfaces and you will have a naming problem (everytime!). We suggest you that you use:
        private var a:String;
public function setA (to:Number):Void {
  this.a = to;
}
public function getA ( Void ):Number {
  return(this.a);
}
        This allows you a pretty setting of vars.
      • Never use public vars. Variables are not controlable, every user of your class can create not planned actions!
      • Your constructor should not do the work for initializing. If you wan't to provide clear constructing use and init method. This allows you even by extending an class to do some operations before you call the super constructor. 
        class A {
  function A () {
    this.init();
  }
  private function init () {
    // Does something.
  }
}
class B extends A {
  function B () {
    this.init();
  }
  private function init() {
    // prepares some stuff
    super.init();
  }
}
      • Try to avoid overloading. If you can't avoid it, use the as2lib overloading handling(see basic concept). So you can pretty documentate youre method.
      • Try to avoid counting of any variables name and dont abbrevate names. All names can have a good name that describes it's value.
      • Don't try to hide mistakes by the user from your class. Throwing Exceptions is always prefered & often faster.
      • Use less bigger interfaces instead of many small, you loose the overview and it's often not clear wheter to use one or more.
      • Try to use the same names for the same things like the others in the project does. This allows all getting faster into the materia.
        Examples:
        owner The owner from an Instance.
        i, j, k Primary, secondary and tertiary counter.
        returnValue Value that gets returned.
        to Setting something to something else. (by using setting & getting)
        e Exception variable. Use the whole name of the excpetiontype if you have more than one. Try to avoid counting.

    • Testcases

      Testcases help us finding errors by refactoring and to notice if all functions are running perfectly. Use testcases for your classes as exactly as possible. Try to find every obscure way to use your class, and if an error occurs that another one notices, add it to the testcase! For internal uses, take the Testcase System defined in the basic concept (see here).

  2. Documentation

    We differ between 3 kinds of documentation (sorted by priority):

    1. Incode documentation

      The most important part of the documentation is "incode documentation". With JavaDoc like Documenation we will have soon a big, detailed documentation for all classes. If you define precise and completly what your method/class exactly does noone will have problems with it.
      That has to be done by the class developer. It's the mightiest tool for our documentation.

    2. Examples

      As second step for showing others how your class works you may provide Examples for your class. Soon you will find here how your examples should be provided.
      Of course, that can be a lot of work, but it helps all understanding your thoughts and the power of your work.

    3. Describing documentation

      As an additional step we will be able to take our concepts for the classes to provide a good overview and the big picture on our homepage, so all customer know how to use our stuff and what is makes possible.
