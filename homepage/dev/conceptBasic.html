Basic Classes (worksheet 0.6)

For the success of a project with ActionScript 2.0 it is necessary to integrate additional functions that are low-level in Flash.

Low-level called in this connection means that many features work in a "usually not suggested" way to handle operations so that other features don't have to use these techniques.
(for example: __proto__, constructor(), ... everything that has a connection with static stuff)

  1. Output handling

    Usually, the output from an application within flash runs over trace();(including reports about uncaught errors). The Problem of this way is that it will only work within an environment where "trace" is supported (only the Flash environment supports this by default). All other cases (within a website, in an executeable file, etc.) don't have a standard output. The output of trace actions is not even guaranteed in the Flash environment, because their export can be switched off(and, of course back on) in the publish settings.

    • Reason

      We support this because we think that output is absolutely necessary(for the user as for the developer) and that it should be possible anywhere.

    • Solution

      To make sure that all output can be handled dynamically within any application(and any player) we create a basic bridge to access the needed functionality.

      This bridge should take all requests and forward it to OutputHandlers. There should be several implementations for several kinds of runntime environments.

      This bridge should also decide wether the current output level(for more details see below) allows the requested action. If not, it does simply nothing.

    • API

      All functions will be accessible through the org.as2lib.basic.Out class with the following static functions: org.as2lib.basic.Out

      • .write([String]):Void

        Simple string output to the standard device (comparable with trace).

      • .error([Exception]):Void

        Exceptions are usually not visible within a .swf (except in the Flash MX 2004 testing environment), this method should print exceptions in a better way.

        Note: The Exception handling Chapter will use this functionality by standard.

      • .log([String]):Void

        Logging often works with files and is more strict, the implementation of the Handler should choose how.

        Note: Outsourcing this code to a seperate Logwriter would be oversized.

      • .setOutputHandler([OutputHandler]):Void

        Usually the "StandardOutputHandler"(which is itself implementing the OutputHandlerInterface) should process all actions, but it will be possible to write custom output handlers for specific needs/various systems.

      • .setOutputLevel([Number]):Void

        The Outputlevel defines which actions are processed and which not. It is designed as a binary flag, you may set it to: 
        0 = 000 = No output.
1 = 001 = processing only exceptions.
2 = 010 = processing only logs.
4 = 100 = processing only writes..
        and each combination: 
        7 = 111 = complete output

        In this way, it's simple to extend it. There will be constants to simplify the access: .DO_EXCEPTIONS // 1 .DO_LOG // 2 .DO_WRITE // 4 .DO_EXCEPTIONS_AND_LOG // DO_EXCEPTIONS | DO_LOG .DO_EXCEPTIONS_AND_WRITE // DO_EXCEPTIONS | DO_WRITE .DO_LOG_AND_WRITE // DO_LOG | DO_WRITE .DO_EVERYTHING // DO_EXCEPTIONS | DO_LOG | DO_WRITE

    • OutputHandler

      Every OutputHandler must implement the OutputHandlerInterface. To provide a fundament there will be an 
      org.as2lib.basic.outputHandler.Trace
      class that simply delegates every output to the trace function. Further classes may be: JavaScript OutputHandler and InScreen Outputhandler.

    • Exceptions

      Exceptions will be thrown if ...
      • no Exception was defined (.error())
      • the given String was empty (.write(),.log())
      • no OutputWriter was available (default value is a "TraceOutputWriter")

  2. Exception handling

    Uncaught exceptions are usually displayed by trace(Error.toString()); (link to Macromedia's Error Class). The result is very minimalistic (if toString is not overridden by the exception's class, only: "Error occured" is displayed). This blocks viewing informations about uncaught exceptions within another than the Flash MX 2004 testing environment.

    • Reason

      The Error class provided by Flash is not really helpful. It doesn't allow you to find an error by simple viewing any stack or even the exception name.

    • Solution

      We will provide a class that extends the basic Error class (to keep in mind that it will be better in future), with an extended functionality.

    • New Features

      • Automized name generation

        To stop that you will have automatically to insert an Exception name class MyError extends Error { public var name = "org.as2lib.basic.WrongArgumentException"; } we will use reflections to get automatically the name of the Exception.

      • Stack

        We will save the previous called action with vars as small implementation of a stacktrace (as long as not more is supported) with the arguments object.
        This will save time in searching "where did this exception occur".

      • Exception Conversion

        If you want to throw an Exception on by a different type when you've catched it there will be a functionality to convert one Exception to another, of course, your are able to cast it but this doesn't change the fields and the name would be the same, you wouldn't have noticed any difference. The conversion should fill the exception additionally with the old Stacktrace and type.

    • API

      For using this exception it's only necessary to extend the basic class. Example:class org.as2lib.basic.WrongArgumentsException extends org.as2lib.basic.Exception { }, all the Rest will be done by the basic class. Methods:

      • .getName():String

        Returns the name from the exception.

      • .getStack():TypedArray(StackObject)

        Returns all stackinformations.

      • .getMessage():String

        on Returns the provided message.

      • .toString():String

        Returns a simple descriptionoutput as ascii graphic.
        --- Exception Occured ---
  type: org.as2lib.basic.exceptions.WrongArgumentsException
  message: You did something wrong.
  by: org.as2lib.basic.StringUtil.trim([String 'home'], [Array 'my', 'yours', 'them']);
      org.as2lib.basic.StringUtil.trimLeft(<unknown>);
        This function also provides itself to Out.exception().

      • .convertTo([Function]):Exception

        Returns a new class filled with the new and old informations. (also see here)

    • Example

      • Enduser

        The enduser will see an exception similar to this by the OutputHandler (formed by the OutputHandler): 
        --- Exception Occured ---
  type: org.as2lib.basic.exceptions.WrongArgumentsException
  message: You did something wrong.
  by: org.as2lib.basic.StringUtil.trim([String 'home'], [Array 'my', 'yours', 'them']);
  formerCall: org.as2lib.basic.StringUtil.trimLeft(<unknown>);

      • Developers

        The developer will have to code this: throw new org.as2lib.basic.exceptions.WrongArgumentsException('You did something wrong.', arguments);

      • ExceptionDeveloper

        The Exceptiondeveloper will have to code an exception like this: class org.as2lib.basic.exceptions.WrongArgumentsException extends Exception { }

  3. Event Handling

    • Problem

      Events within Flash can cost much performance and are within User-Interfaces existential. Most people use the inbuild ASBroadcaster or the Eventdispatcher from Macromedia for this solution. The problem within this solution is that you can't control, whether an Handler for an event exists or not. If you do it within a script you may loose a lot of performance.

      Several Problems:
      • The Objectdeveloper has to define which events are supported to listen to.
      • The Listenerdeveloper has to define all events.
      • The Objectdeveloper must have a possibility to fire events.
      • The Objectdeveloper may add eventinformations to the event.
      • Support for Standardevents within an Object might be senseful (internal call)
      • Blocking standard events: In an Eventcase where you have only one listener and an standard-Action, the listener should be able (by returning false) to prevent that the standardaction will be used.

        Note: This may also be with more than one listener.

    • Reason

      We support Eventhandling because its a Keyfeature. Using Macromedia's inbuild class (ASBroadcaster) is unstrict (that means open for good an bad usage). And the Eventdispatcher is not "free". You only may use it when you bought Flash and it doesn't fit into the needs from the developers.

    • API

      We act at different fronts:

      • Basic Class

        The basic class ist the main class for eventhandling. It will be provided within: 
        org.as2lib.basic.EventHandler
        This class should be extended if you want to access all functionalities:
        • .addListener([Listener]);
          Add's an Listener (to all event types) to the Object.
        • .removeListener([Listener]);
          Remove's an added Listener.
        • .getListeners():TypedArray([Listener]);
          Returns all Listeners within an Typed Array.
        • .dispatchEvent([Event]);
          Dispatches an Event for all Listeners.

          Note: The Object itself will be an Listener too.

        • [static] .logEvents([boolean]);
          Defines whether all Events (from all Classes) should be logged or not. org.as2lib.basic.EventHandler.logEvents(true);

        Note: This class is not specified as abstract class, to use it if you can't extend it.

        import org.as2lib.basic.* class MyClass extends SomethingDifferent { private var eventHandler:EventHandler; function MyClass () { eventHandler = new EventHandler(); } }

      • Event abstract class

        There will be an Eventabstractclass to define which event has to be called.
        Each Event will have to extend this class!
        The only field from 
        org.as2lib.basic.Event
        is [static] "eventName" that has to be set (which event is fired), different vars may be set too, but they will be specific for this event.

      • Basic Interface

        The only use from this Interface, found in 
        org.as2lib.basic.EventListenerInterface
        is to make sure an Listener is used an not any object.

      • Generic EventHandling

        We won't support generic eventhandling, the ASBroadcaster does this ugly job faster.

    • Example

      • ObjectDeveloper

        1. Class where listener should be addable
          import org.as2lib.basic.* class MyClass extends EventHandler { private function onEnterframe (e:MyEventHandler) { } public function addListener (l:MyEventInterface) { super.addListener(l); } public function gotoFrame (frame:Number) { this.dispatchEvent(new MyEventHandler(frame); } }
        2. Returned Eventhandler
          import org.as2lib.basic.* class MyEventHandler extends EventHandler { public var eventName:String = "onEnterFrame"; public var frameNumber:Number; public function MyEventHandler (frameNumber) { this.frameNumber = frameNumber; } }
        3. Eventinterface
          import org.as2lib.basic.* class MyEventInterface extends EventInterface { public function onEnterframe (e:MyEventHandler):Void; }

      • ListenerDeveloper

        1. Listenercreation
          import org.as2lib.basic.* class MyEventListener implements MyEventInterface { public function onEnterframe (e:MyEventHandler){ Out.write('entered Frame number '+e.frameNumber); } }
        2. Listenerregistration
          ... var myObject:MyClass = new MyClass(); myObject.addListener(new MyEventListener()); ...

  4. Typed Array

    A typical problem using as2 is that an array may have more types than one, this should never happen, especially when you want to publish your classes (within a team or generally), because other people can easily create chaos if they don't watch out.

    • Reason

      We don't think flash will ever support this because Ecma Script has no support for this, it's needed almost everytime, so we decided to work on that.

    • Solution

      The solution is preety simple: We take an array to an "not dynamic"-class and change all add-functionalities to type-matching functionalities.

    • API

      There will be one basic class that can be used as is or be extended:
      org.as2lib.basic.TypedArray
      You have to add an constructor type to that class: new org.as2lib.basic.TypedArray(String) Methods from TypedArray (differing to the Array Methods):

      • .push()

        Push will throw an exception if you want to add a value with the wrong type.

      • .concat()

        Concat will throw an exception if you try to use values with the wrong type.

      • .unshift()

        Push will throw an exception if you want to unshift values with the wrong type.

      • .slice()

        Slice will return an TypedArray instead of a normal Array.

      • .splice()

        Splice will return an TypedArray instead of a normal Array.

      • .setField([String], Value);

        Will add the content to the array(and throw an exception if the type doesn't match).

      • .getField([String]);

        Will return the content from the field by the name.

      • .getType()[Object];

        Will return the type that has been set.

    • Exceptions

      Exceptions will be thrown if ...
      • you forget setting an type within the constructor.
      • you try to add something to the array that doesn't match with the array's typedefinition.

    • Problems

      1. TypeInsurance

        instanceof has a "bug" or its a flash specifica that i not really understand, this:
        [ClassA extends ClassB]
(new ClassA instanceof ClassB)
        is true!
        That means we have to use an second check that this object really doesn't extend any class (even no superclass).

      2. Flash Basic Functionality

        In Flash its possible to access every method and content with []. a = new NotDynamicClass(); a["monkey"] = "true"; To prevent this we will not extend the Array, this class will be an bridge to the array.

With the path and "eval" i'm able to fetch everything else.

  7. Cloning

    Cloning like in Java would run with a simple Interface that defines "this class is cloneable". Flash offers a way to clone a class generic.

    • Reason

      Cloning is something everyone needs sometimes, an support would make most times the work easier.

    • Solution

      Within the util class 
      org.as2lib.basic.Clone
      we'll provide simple cloning. The Basic Class will also provide access.
      Generally this will work by creating a new instance from the class and setting all fields to the values from the old field.

    • API

      • .clone()

        Will return a new object from the same class with the same values.

      • .cloneCompletly()

        This will return recursivly from all Objects a clone.

    • Problems

      The worst Problem with cloning is the fact that some classes throw exceptions if not the correct Value has been set.
      This will be hidden by try {}.

  8. Overloading

    Overloading is using different arguments for one Method. Overloading is only manually possible within as2.

    • Reason

      Of course we all should avoid overloading as good as possible, it's not beautiful that as2 doesn't support it but its logical and we have all to live with it. The "standard-way" of using overloading is pretty ugly combined with a lot of code. So we decided to support this feature for people that want to use overloading.

    • Solution

      We support overloading with simple functionalities accessed by an util class or within the basic class.

    • API

      The primary methods should be available within the util class: 
      org.as2lib.basic.Overloading

      • .overload(applyTo:Object, arguments:Arguments, handler1:OverloadHandler, ... , handlerN:OverloadHandler)

        Checks the arguments if it matches to one handler and then executes the handler with this arguments by the Object(applyTo).

      • .createOverloadHandler(arguments, functionString)

        Creates an new OverloadHandler (factory), for easier access.
      There will be additionally a class for the Handler: 
      org.as2lib.basic.OverloadHandler
      This has to be initialized with new OverloadHandler(arguments:Arguments, function:String);

      • .matches(arguments)

        Returns true if the arguments match to the Handlers argument.

      • .getFunctionString

        Returns the functionstring set to the Overloadhandler.

      Note: applyTo is only needed by accessing the utils, access within the basic class will not need that.

    • Exceptions

      Exception will be thrown if ...
      • the arguments dont match any handler.
      • there are two handler with the same arguments array

    • Example

      As example for using Overloading:

      • Within the basic class:

        public function doSomething() { overload ( arguments, createOverloadHandler([String, Number], "doSomethingWay1"), createOverloadHandler([String], "doSomethingWay2") ); }

      • Using the util class:

        import org.as2lib.basic.BasicUtil; ... public function doSomething() { BasicUtil.overload ( this, arguments, BasicUtil.createOverloadHandler([String, Number], "doSomethingWay1"), BasicUtil.createOverloadHandler([String], "doSomethingWay2") ); }

    • Possible future addition

      It could be very possible to use wildcards within the Overloadhandler.

  9. Testcases

    A Testcase is a Class/Method that checks a different method if it works correctly. A Developer has to write a Testcase alone for his class, a Testcase system has to provide the functionality for automated testing.

    • Reason

      After a lot discussion we decided to use our own test system instead of the as2unit, specially because they use components, don't support a lot of methods and are not opensource (they got opensource, but the source is still not available).

    • Solution

      We will provide our own Testcase Class to extend, this should (except the run call and additional Methods) not differ from as2unit in most cases.

    • Differences

      • Running code

        To run Testcases you should not have to do two different things(as as2unit has). Developers should be able to start a testcase within a call(not add a whole component to the stage). This should work with <basicClass>.run("<path>") this way is usual and simple but within flash it doesn't compile the testcase if you use a string. So we will provide a second access to run a testcase seperatly with <basicClass>.runClass(<path>) so the Class itself will be compiled because it is defined as is within the call.

      • Handling Output

        The Output Handling should run over the standard OutputHandling. Simple Text Output to Asccii graphics. Another difference to other testsystems should be defined by displaying more details within the result (like which testcases were found and which work correctly), mostly because it's often not clear whether a testcase is found or not.
