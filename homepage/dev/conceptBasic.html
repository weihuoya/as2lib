Basic Classes (worksheet 0.5)

For the success of a project with ActionScript 2.0 it is necessary to integrate additional functions that are low-level in Flash.

Low-level called in this connection means that many features work in a "usually not suggested" way to handle operations so that other features don't have to use these techniques.
(for example: __proto__, constructor(), ... everything that has a connection with static stuff)

  1. Output handling

    Usually, the output from an application within flash runs over trace();(including reports about uncaught errors). The Problem of this way is that it will only work within an environment where "trace" is supported (only the Flash environment supports this by default). All other cases (within a website, in an executeable file, etc.) don't have a standard output. The output of trace actions is not even guaranteed in the Flash environment, because their export can be switched off(and, of course back on) in the publish settings.

    • Reason

      We support this because we think that output is absolutely necessary(for the user as for the developer) and that it should be possible anywhere.

    • Solution

      To make sure that all output can be handled dynamically within any application(and any player) we create a basic bridge to access the needed functionality.

      This bridge should take all requests and forward it to OutputHandlers. There should be several implementations for several kinds of runntime environments.

      This bridge should also decide wether the current output level(for more details see below) allows the requested action. If not, it does simply nothing.

    • API

      All functions will be accessible through the org.as2lib.basic.Out class with the following static functions:

      • .write([String]):Void

        Simple string output to the standard device (comparable with trace).

      • .error([Exception]):Void

        Exceptions are usually not visible within a .swf (except in the Flash MX 2004 testing environment), this method should print exceptions in a better way.

      • .log([String]):Void

        Logging often works with files and is more strict, the implementation of the Handler should choose how.

      • .setOutputHandler([OutputHandler]):Void

        Usually the "StandardOutputHandler"(which is itself implementing the OutputHandlerInterface) should process all actions, but it will be possible to write custom output handlers for specific needs/various systems.

      • .setOutputLevel([Number]):Void

        The Outputlevel defines which actions are processed and which not. It is designed as a binary flag, you may set it to: 
        (0) 000 = No output.
1 = 001 = processing only exceptions.
2 = 010 = processing only logs.
4 = 100 = processing only writes..
        and each combination: 
        (7) 111 = complete output

        In this way, it's simple to extend it. There will be constants to simplify the access: .DO_EXCEPTIONS // 1 .DO_LOG // 2 .DO_WRITE // 4 .DO_EXCEPTIONS_AND_LOG // DO_EXCEPTIONS | DO_LOG .DO_EXCEPTIONS_AND_WRITE // DO_EXCEPTIONS | DO_WRITE .DO_LOG_AND_WRITE // DO_LOG | DO_WRITE .DO_EVERYTHING // DO_EXCEPTIONS | DO_LOG | DO_WRITE

    • OutputHandler

      Every OutputHandler must implement the OutputHandlerInterface. To provide a fundament there will be an 
      org.as2lib.basic.outputHandler.Trace
      class that simply delegates every output to the trace function. Further classes may be: JavaScript OutputHandler and InScreen Outputhandler.

    • Exceptions

      Exceptions will be thrown if ...
      • no Exception was defined (.error())
      • the given String was empty (.write(),.log())
      • no OutputWriter was available (default value is a "TraceOutputWriter")

  2. Exception handling

    Usually Exceptions within Flash get displayed with the trace(Error.toString()). The Result is very rare (if nothing defined only: "Error occured" will be displayed) This blocks exception-viewing within another environment than Flash MX 2004.

    • Reason

      The Error Class that Flash uses ist in best mind not useable. It doesn't allow you to find an error.

    • Solution

      We will provide a class that extends the basic Error class (to keep in mind that it will be better in future), with new features.

    • New Features

      • Automized name generation

        To stop that you will have automatically to insert an Exception name class MyError extends Error { public var name = "org.as2lib.basic.WrongArgumentException"; } we will use reflections to get automatically the name of the Exception.

      • Stack

        We will save the previous called action with vars as small implementation of a stacktrace (as long as not more is supported) with the arguments object.
        This will save time in searching "where did this exception occur".

      • Exception Conversion

        If you want to throw an Exception on by a different type when you've catched it there will be a functionality to convert one Exception to another, of course, your are able to cast it but this doesn't change the fields and the name would be the same, you wouldn't have noticed any difference. The conversion should fill the exception additionally with the old Stacktrace and type.

    • API

      For using this exception it's only necessary to extend the basic class. Example:class org.as2lib.basic.WrongArgumentsException extends org.as2lib.basic.Exception { }, all the Rest will be done by the basic class. Methods:

      • .getName():String

        Returns the name from the exception.

      • .getStack():TypedArray(StackObject)

        Returns all stackinformations.

      • .getMessage():String

        on Returns the provided message.

      • .toString():String

        Returns a simple descriptionoutput as ascii graphic.
        --- Exception Occured ---
  type: org.as2lib.basic.exceptions.WrongArgumentsException
  message: You did something wrong.
  by: org.as2lib.basic.StringUtil.trim([String 'home'], [Array 'my', 'yours', 'them']);
      org.as2lib.basic.StringUtil.trimLeft(<unknown>);
        This function also provides itself to         Out.exception().

      • .convertTo([Function]):Exception

        Returns a new class filled with the new and old informations. (also see here)

    • Example

      • Enduser

        The enduser will see an exception similar to this by the OutputHandler (formed by the OutputHandler): 
        --- Exception Occured ---
  type: org.as2lib.basic.exceptions.WrongArgumentsException
  message: You did something wrong.
  by: org.as2lib.basic.StringUtil.trim([String 'home'], [Array 'my', 'yours', 'them']);
  formerCall: org.as2lib.basic.StringUtil.trimLeft(<unknown>);

      • Developers

        The developer will have to code this: throw new org.as2lib.basic.exceptions.WrongArgumentsException('You did something wrong.', arguments);

      • ExceptionDeveloper

        The Exceptiondeveloper will have to code an exception like this: class org.as2lib.basic.exceptions.WrongArgumentsException extends Exception { }

  5. Event Handling

    Eventhandling within as2 should be a two sided sword. On the one hand it is necessary to have an easy handling for an event itself. On the other hand is generic eventhandling a problem because you can't guarantee that all event's fired reach the final client.

    Generic Eventhandling

    For generic eventhandling there will be a (abstract) class org.as2lib.basic.EventableElement extends BasicObject to handle general events. This class will not have ANY public method. But this class should be based on the EventDispatcher by Macromedia.

    This class will provide the functionality to register and fire generic events by .addEventListener(object, "event"); .removeEventListener(object, "event");

    Eventrule for Using

    For cute Handling of this code the Workflow will have to look like this:
    1. The Classwriter should define an Interface where all Event's are defined (how should elseway be sure that you implement Events?) interface MyEventInterface { function onClick(e:MyEventObject):Void; function onRelease(e:MyReleaseEventObject):Void; }
    2. The Classwriter will have to define methods to add an Object using this Events. class MyClass { public function addMyEventListener(obj:MyEventInterface) { this.addEventListener(obj, "onClick"); this.addEventListener(obj, "onRelease"); } }
    3. The User from the Code will have to take a look at the Interface. class MyEventListener implements MyEventInterface, AnotherEventInterface { public function onClick(e:MyEventObject):Void { // does something } public function onRelease(e:MyEventObject):Void { // does something different } // and other events }
    4. Then the user add's his Eventhandler to the Object new MyClass().addMyEventListener(new MyEventListener());

    Why so complicated?

    This is actually the best way to ensure that a function is really available and it finally reduces the operation speed of eventhandling by a better Code Handling. Of Course this way isn't perfect. It will maybe get changed within a future worksheet by a better idea, but actually i see no different possibility.

    Different access

    Well, first previewable Problem will be that one class is not able to extend this Eventhandler, because it has already extended on class. The simplest way to allow this way is to get an Eventhandler as instance. function MyClass () { this.eventHandler = new org.as2lib.basic.EventHandler(); } And the rest works as used only with "this.eventhandler.addEventListener()" instead of "this.addEventListener()". The only problem ist that we have already the abstract class, so this class has to be an own extension:class org.as2lib.basic.EventHandlerUseable extends org.as2lib.basic.EventHandler /* Names not fixed */ { public function addHandler( ... ) { return(this.addEventHandler( ... )); } ... // or similar }

    Note: this Issue might change a little. (naming and real functionality)

  6. Typed Array

    A typical problem using as2 is that an Array might have one or more types within an Array, in most cases, this is very problematic, especially when you want to publish your classes (within a team or generally), because other people can easily create chaos when they don't watch.

    Where to find?

    Of course we need a Class you will find it at org.as2lib.basic.TypedArray. This class will use the as2 Array class and uses all Methods like Array as a Bridge. It should not be possible to directly set/get a Value like # myArray = new Array(); myArray["name"] = "bunny"; because the Compiler would not check it. We cant prevent this (the compiler completly blows it). But at least we can bind ( by not using a dynamic class, like Array is) that myArray.name = "bunny" will not work.

    Typedefinition

    The type of an Array should be defined within the constructor. Infact the Type should be the hole lifetime the same.new TypedArray(org.as2lib.basic.Exception)

    Typeinsurance

    We have a "not-dynamic" Class. So there must be methods for adding a value within this array with a name.
    • validation: public function checkType(value) { if(value instanceof this.type) { return(this.type); } else { throw new org.as2lib.basic.exceptions.WrongTypeException("The given Value doesn't match to the Array's type"); } }
    • setting: public function setValue(name:String, to) { this.internalArray[name] = checkType(to); }
    • getting: public function getValue(name:String) { return(this.internalArray[name]) }

    Bridge Functionality

    Even the validation and the setting/getting of values are almost simple bridge functionalities. Different Methods like "sort, push, pop, ...." can surly be simple ported.

  7. Cloning

    Cloning like in Java would run with a simple Interface that defines "this class is cloneable"(this will also exist in our case). Of course this interface should be shurly used by us too but also it as2 allows us a generic clone.

    Idea

    Often it's complicating that the compiler won't check all possibilities and no exceptions won't be thrown (even if the developer does really bad stuff). But it gives us the possibilty for simple using the objects constructor and proto to make a real clone with that. One "for(in)" loop could handle all that.

    Location

    Of course it's basic functionality like Reflections so this should be also in the "BasicClass" that all other extend as ".clone()" Method.

    Access points

    The code/handling should be in a Util class "org.as2lib.basic.Helper"(or something like that)

    Complete cloning

    Sometimes it is a real problem that cloning only clones one level. There should be additionally a Method to clone the complete tree ".cloneComplete" which is defined as "Chain of Responsibility" (Design Pattern) this simply uses ".cloneComplete" recursivly (okay it's a little more difficult).

    Note: Interfaces (specially small one) make problems with casting, therefore this won't be available within an interface.

  8. Overloading

    Of course we all should avoid overloading as good as possible, it's not beautiful that as2 doesn't support it but its logical and we have all to live with it. The "standard-way" of using overloading is pretty ugly combined with a lot of if's and code. So we decided to support this feature.

    Target

    Maybe the future presets for us a as2 that supports overloading, but not now. But we should be prepared for that, so each possibility should be defined in its own function: public function doSomethingWay1() { } public function doSomethingWay2() { } In this case it will be nice in future to change it to a supported overloading. The second thing is that this can be well documented. But this means a Separation for this overloading would look like this: public function doSomething() { if(arguments[0] instanceOf Home) { doSomethingWay1(arguments[0], arguments[1]); } else { doSomethingWay2(arguments[0]); } } This is everytime a new challenge for the developer and may cause a lot of problems. Because if you want to check all you have unreadable code, a source for errors and maybe unchecked causes (what if the arguments don't match any case?!). Here is where we want to bring our part into the game. We should provide a functionality that handles these switch automatically. This functionality should be provide the same way like the Reflections: within the basic class and as util. There you should only use a predefined method for that: public function doSomething() { overload ( arguments, createOverloadHandler([String, Number], "doSomethingWay1"), createOverloadHandler([String], "doSomethingWay2") ); }

    Note: This is not the final code! The Syntax might change!
    Note 2: "this." Keyword is not important, it will be set by the compiler.
    Note 3: createOverloadHandler returns an OverloadHandler Instance from the Utils Class.

    Wrong calls

    A wrong call is a wrong call if it's size and it's types have no match. In this case, an exception will be thrown.

    Public Access

    The public Access over the Util Class will look like:import org.as2lib.basic.BasicUtil; ... public function doSomething() { BasicUtil.overload ( this, arguments, BasicUtil.createOverloadHandler([String, Number], "doSomethingWay1"), BasicUtil.createOverloadHandler([String], "doSomethingWay2") ); }So the complete Handling will work even if the basic Class can't be extended.

    Possible Extension

    There could be a possible Extension to insert Wildcards where an (un)-limited part of vars from a type can follow. (future release)

  9. Test Cases

    After a lot discussion we decided to use our own test system instead of the as2unit, specially because they use components and don't support a lot of methods.

    What are Testcases

    A Testcase is a Class/Method that checks a different method if it works correctly. A Developer has to write a Testcase alone for his class, a Testcase system has to provide the functionality for automated testing

    What we do like other systems

    We have (like all other System too) a basic Class that should be extended. This class will support different functionalities to compare values and define wether a testcase failed or not (for example: assertEquals(), assertTrue() etc.). For all this functionality there should be a inverse functionality to check the difference (example: assertNotEquals(), assertFalse(), etc.).
    Usually a TestcaseSystem works with overloading, but in fact, we decided to use this as rare as posible so each method should have an additional method to provide a message to the failure (for example: assertTrueWithMessage(), assertFalseWithMessage(), etc.).

    Note: with as2unit you have also to make different notations to the testcase itself, we will avoid this.

    Running Code

    To run Testcases you should not have to do two different things(as as2unit has). Developers should be able to start a testcase within a call(not add a whole component to the stage). This should work with <basicClass>.run("<path>") this way is usual and simple but within flash it doesn't compile the testcase if you use a string. So we will provide a second access to run a testcase seperatly with <basicClass>.runClass(<path>) so the Class itself will be compiled because it is defined as is within the call.

    Handling Output

    The Output Handling should run over the standard OutputHandling. Simple Text Output to Asccii graphics. Another difference to other testsystems should be defined by displaying more details within the result (like which testcases were found and which work correctly), mostly because it's often not clear whether a testcase is found or not.

  10. Regulary Expressions

    Regulary Expression might count to the StringUtils but in my opinion Regulary Expressions can handle a lot and improve the work well, so i put it to the Basic Concept.

    What are Regulary Expressions?

    Regulary Expressions are a simple but mighty way to search within Strings. JavaScript supports Regulary Expressions well and i got no idea why flash not.

    How will they be supported?

    Support will come within a Util Class. One Accesspoint (at least) will handle a Regulary Expression. Further details will come during real concept and Worksheet for this.

    Note: This has the lowest priority.
